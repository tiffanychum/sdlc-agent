## Part 1 — Top 3 Python Async Programming Best Practices (2025–2026)

### ✅ Best Practice #1: Prefer `asyncio.TaskGroup` over bare `asyncio.gather()`
_Introduced in Python 3.11 (PEP 654) — now the community standard as of 2026_

`asyncio.TaskGroup` provides **structured concurrency**: tasks are scoped to a block,
exceptions from any child task cancel all siblings and re-raise immediately — no silent
failures. `asyncio.gather()` can swallow exceptions or leave tasks dangling if not
carefully managed.

```python
# ✅ Modern (Python 3.11+)
async with asyncio.TaskGroup() as tg:
    task_a = tg.create_task(fetch_data())
    task_b = tg.create_task(process_items())
# Both tasks are guaranteed complete or cancelled here

# ⚠️ Old pattern — still works but error-prone
results = await asyncio.gather(fetch_data(), process_items(), return_exceptions=True)
```

---

### ✅ Best Practice #2: Always set timeouts with `asyncio.timeout()` (not `wait_for`)
_`asyncio.timeout()` context manager — stable since Python 3.11_

Never let async I/O operations run indefinitely. `asyncio.timeout()` is the modern,
composable replacement for `asyncio.wait_for()`, works cleanly with `TaskGroup`, and
avoids the pitfall of `wait_for` cancelling too aggressively.

```python
# ✅ Modern
async with asyncio.timeout(10.0):
    result = await some_network_call()

# ⚠️ Old pattern — still common but less composable
result = await asyncio.wait_for(some_network_call(), timeout=10.0)
```

**Applies directly to this codebase:** `src/llm/client.py` and `src/mcp_servers/web_server.py`
should be audited to confirm timeouts are set on all outbound HTTP/LLM calls.

### ✅ Best Practice #3: Use `anyio` or `asyncio` primitives for cancellation-safe cleanup — avoid bare `try/finally` around `await` calls without shielding
_Widely adopted in 2025–2026, especially in production async services_

Cancellation in asyncio propagates via `CancelledError`. Code that does cleanup in
`finally` blocks containing `await` calls may silently suppress cancellation or leave
resources in an inconsistent state. The best practice is:

- Use `contextlib.asynccontextmanager` for resource management.
- Use `asyncio.shield()` **only** where truly necessary (e.g., flushing a DB write).
- Prefer libraries that respect structured cancellation (e.g., `anyio`, `httpx`, `aiofiles`).

```python
# ✅ Safe async resource cleanup
@contextlib.asynccontextmanager
async def managed_connection():
    conn = await db.connect()
    try:
        yield conn
    finally:
        await asyncio.shield(conn.close())  # shield only the critical close
```
